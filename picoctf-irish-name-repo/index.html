Pico CTF - Irish-Name Repo

January 04, 2021

Pico CTF has been around for awhile and it seems to be more approachable one amongst other platforms alike. Different from Hackthebox or Tryhackme (where “pwn a box” usually means chaining security vulnerabilities altogether towards finding the flags), on every picoCTF challenge, you just get an URL where there’s a small puzzle to be solved. Most of the time it involves one specific type/class of security flaw - like XSS or SQLi for instance - and that’s about it. Also, you don’t really need VPN’s or any additional setup other than creating an account. Simple and easy.

Irish-Name Repo challenges

I was basically skimming through the Web Exploitation category when I came across Irish-Name Repo. The nice thing about this pick is that it is a composed challenge, each one slightly more difficult than the previous. It was apparently part of the picoCTF 2019, so not that old either. Lets get to it.

Irish-Name Repo 1

So right after selecting the first challenge, it displayed a modal with the URL and a small description of what needed to be achieved:

Goal: Bypass a login page

Clicking on the link tool me to a page listing some people (presumably Irish). I could see their pics, along with their names and a short description. On the left, there was a menu (you have to click on the hamburger icon) that as soon as it was displayed, presented me with three options: Close Menu, Support and Admin Login. There was also a pagination at the very bottom but that didn’t really work. Pressing CTRL+U and quickly checking the source code gave me nothing other than a pretty standard page html page.

The first link on the menu did exactly what it said it would: It closed the menu. The second item got me to a forum sort of page where there was a discussion about irish people. First comment was a giveaway:

“Hi. I tried adding my favorite Irish person, Conan O’Brien. But I keep getting something called a SQL Error.”

After that, the obvious place to go was the admin login. The login was pretty standard and the first thing I did was to add a single quote ' to both Username and Password fields, expecting the application to throw me a SQL error. Instead, I got a “Login failed” response. Small tweak, this time adding bob' in the username field and which threw me a 500 Internal Server Error. That was the indication this actually broke the SQL query.

After that, I went for the obvious bob' or 1=1-- on the Username. This got me logged in and along with that, the first flag.

Explanation

This might be the most basic SQL injection technique - and most well known - to bypass login pages. It has been around for quite some time and consists in two things:

  • The fact that the input isn’t sanitized and injects whatever you passed in directly into the query;
  • The fact that by manipulating the query, it allows you to disregard the first piece of the condition by creating a second one that always evaluate to true;

The standard query to retrieve and validate wether an user actually exists could be something like this:

SELECT FROM users WHERE username='bob' AND password='p4ssw0rd';

Getting a response from this query means the user sent by whoever is trying to log in actually exists. So now, take a moment to double-check what we just did when the bob' or 1=1-- was introduced as a username. I will rewrite the SQL so you can see for yourself:

SELECT FROM users WHERE username='bob' or 1=1--' AND password='p4ssw0rd';

And now to sum it up:

  • We first introduce a ', which in the context of this query, sets the boundary of that string - bob - informed
  • Then, we add our condition 1=1 that happens to always evaluate to true. This for instance, could also be replace by 2 > 1 or 'a'='a'

  • Last but not least: we add a -- which stands for comment in SQL, and this tells the interpreter to ignore the remaining pieces of this query

    The outcome is a query that evaluates to true allowing me to login without an actual valid password. Hopefully that made sense.

Irish-Name Repo 2

The second challenge landed me back to the same page. This time however, the description hinted the SQL had been improved or in other words, my first bypass was now discarded. Lets have a look.

Goal: Bypass the enhanced login page

I went back to the admin page and tried the same payload I used before admin' or 1=1--. What happened this time however, is our SQLi attempt gets detected right away as shown down below:

After playing with the input for a second, I noticed what was giving away our SQLi was in fact the or piece as part the username. So everything with or in the condition would trigger the WAF. I then, removed the or 1=1-- altogether and replaced it with just admin'--. Boom, got the second flag.

Explanation

So in this case the validator was only looking for or as part of the query, and I needed to find away to accomplish the same result without it. What I did then was take into account the fact that I knew beforehand the admin was a valid user - he was posting in the forum page - and that’s all. This is how the query looks like:

SELECT FROM users WHERE username='admin'-- AND password='p4ssw0rd';

This again, returned a response which is enough for me to get new the flag.

Irish-Name Repo 3

Until now, this had been a walk in a park. Even if you are not familiar with SQLi techniques, just by looking at payloads out there you would eventually be able to log in. The last piece of this challenge requires a little investigation nonetheless.

Goal: Bypass the latest login page which now only displays the password input field.

I once again, found myself staring at the admin page just to find out this time, there was only a password field being displayed. Since I didn’t know any passwords and there was no way to know them just like I did with usernames, I stopped to understand my options for a second. Just out of curiosity, I decided to throw admin'-- just to get that out of the day and that obviously didn’t work. I then, pressed CTRL+U to see the page’s source code. Carefully checking the form to submit the request, I noticed a hidden field called debug. Now with the developer tools, I changed its value from 0 to 1, password to abc and again, submitted the request.

The result can be seen here:

After that, I’ve tried a few other passwords without any lucky until I finally figured out what was happening. I then, sent the payload ' be 1=1-- and got the last missing flag. Over and out.

Explanation

Here’s what happened: The result from abc was nop as you could see. And a is one letter of distance from b while c are two letters from a right? The same applies to our password response nop. From that point on, it was pretty easy to tell… On the backend, somewhere somehow to each letter of our password, the position is shifted by 12 chars to the right from its original position.

So lets say we send car as our password. The result is going to be:

  • First letter is c: shift 12 positions away and you’ll get the letter p.
  • Second letter is a: shift 12 positions away and you’ll get the letter n.
  • Last letter is r: shift 12 positions away and you’ll get the letter f (its circular, so once it reaches the letter z it starts over)

Dope, but again.. what do we do with that information now? Well, how about crafting a payload that after converted, translates its final result into or 1=1--? That could work. So applying the same idea to the new password, that would be be 1=1. This is how we get the last flag.
